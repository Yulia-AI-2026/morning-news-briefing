# 工作流详解 · Workflow Details

## 完整链路

```
07:30  cron 触发
  │
  ├─ ① 获取天气
  │    wttr.in → 温度/湿度/风速/天气描述
  │
  ├─ ② 读取日记
  │    从前一日日记提取：状态/计划/关注点 → 搜索关键词
  │
  ├─ ③ 动态选题
  │    不是固定赛道。从日记长出的需求决定今天搜什么。
  │    参考池：AI科技 / 上海本地 / 内容创作 / 心理健康 / 女性议题 / 消费经济 / 当日要闻
  │
  ├─ ④ 搜索
  │    web_search × 多个关键词（Tavily 引擎）

  ├─ ⑤ 筛选（核心关卡）
  │    每条必须回答：为什么栀安需要知道这个？
  │    公式：本地+即时 × MEMORY当前方向挂钩 × 行动入口
  │    时效性铁律：只选3天内的新闻
  │    宁缺毋滥：凑不够5条就少写
  │
  ├─ ⑥ 生成 Markdown
  │    固定格式，不可偏离（见 SKILL.md 格式铁律）
  │    写入 /tmp/morning_news.md
  │
  ├─ ⑦ 邮件发送
  │    Python 脚本 → QQ SMTP → 3865560266@qq.com
  │
  ├─ ⑧ 微信保底
  │    Nanobot message 工具 → 微信通道
  │    无论邮件成功与否，至少一个通道送达
  │
  └─ ⑨ 清理
       rm /tmp/morning_news.md
```

## 关键决策点

### 为什么从日记提取话题而不是固定赛道？

v1.0-v2.1 使用固定五大赛道。问题：
- 机械扫描，话题与用户当天状态脱节
- 用户前一天在焦虑工作、早报却在讲猫咪救助

v3.0（2026-05-11）改为动态提取：
1. 先读前一天日记
2. 把当前状态/计划/关注点提炼成搜索关键词
3. 再结合话题池搜索

**效果**：新闻从"AI 觉得你该看的"变成"从你生活里长出来的"。

### 为什么双通道？

邮件是主通道（格式完整、可存档），微信是保底（即时触达）。单通道和单点故障是同义词。
